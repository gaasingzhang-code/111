# 边界用例分类体系

用于 Agent 在需求分析阶段系统化枚举边界用例。优先级 = 触发频率 × 影响程度。

---

## 类别一：边界值 (Boundary Values)

**触发问句**：输入达到最大/最小值时会发生什么？

| # | 场景模式 | 示例 |
|---|---------|------|
| BV-1 | 数值输入为 0 / 负数 / 极大值 | 购物车数量输入 -1 |
| BV-2 | 字符串长度为 0 / 最大长度 / 超长 | 用户名输入 500 字符 |
| BV-3 | 列表为空 / 单条 / 超大列表 | 搜索结果 0 条 / 10000+ 条 |
| BV-4 | 时间边界（跨天/跨月/跨年） | 跨月查询账单 |
| BV-5 | 金额精度边界 | 支付金额 0.001 元 |

---

## 类别二：并发与竞态 (Concurrency & Race Conditions)

**触发问句**：两个操作同时发生时系统行为是否确定？

| # | 场景模式 | 示例 |
|---|---------|------|
| CR-1 | 同一资源被同时修改 | 两人同时编辑同一文档 |
| CR-2 | 库存/限额并发扣减 | 最后一件商品被两人同时下单 |
| CR-3 | 重复提交 | 用户双击提交按钮 |
| CR-4 | 异步回调时序不确定 | 支付回调晚于超时取消 |
| CR-5 | 定时任务与用户操作并发 | 凌晨结算时用户正在下单 |

---

## 类别三：状态转换 (State Transitions)

**触发问句**：从任意状态触发任意操作的结果是否定义完整？

| # | 场景模式 | 示例 |
|---|---------|------|
| ST-1 | 非法状态转换 | 已取消订单再次支付 |
| ST-2 | 中间态下的用户操作 | 支付处理中用户关闭页面 |
| ST-3 | 终态后的操作 | 已完成订单申请退款 |
| ST-4 | 状态回退 | 审核通过后撤回重审 |
| ST-5 | 超时自动状态变更 | 30分钟未支付自动取消 |

---

## 类别四：输入校验与异常输入 (Input Validation)

**触发问句**：异常输入能否优雅处理而不崩溃？

| # | 场景模式 | 示例 |
|---|---------|------|
| IV-1 | SQL/脚本注入 | 搜索框输入 `<script>alert(1)</script>` |
| IV-2 | 特殊字符 / Emoji / 零宽字符 | 姓名含 Emoji: "张😀三" |
| IV-3 | 类型不匹配 | 期望数字传了字符串 |
| IV-4 | 必填字段为空 / 全部字段为空 | 提交空白表单 |
| IV-5 | 文件上传异常 | 上传 0 字节文件 / 伪装扩展名 |

---

## 类别五：异常恢复 (Error Recovery)

**触发问句**：异常发生后用户能否恢复数据和工作状态？

| # | 场景模式 | 示例 |
|---|---------|------|
| ER-1 | 网络中断后恢复 | 填写表单时断网，恢复后内容是否保留 |
| ER-2 | 服务超时/500 错误 | API 超时后用户操作状态 |
| ER-3 | 依赖方数据异常 | 第三方物流返回空 tracking_number |
| ER-4 | 用户中途退出后回来 | 关闭 App 24 小时后重新打开 |
| ER-5 | 回滚/补偿失败 | 退款失败后的处理 |

---

## Agent 使用说明

1. 在 Phase 2（分析阶段）逐类过这 5 个类别
2. 每个类别至少产出 1 条适用于当前项目的边界用例
3. 输出格式：`[分类-编号] [场景描述] → [预期行为] [来源：分析推断/访谈提及-角色]`
4. 如果项目在某类别下确实无风险，标注 `[分类-无风险]` 并附理由（如"无并发场景"）
5. 优先级标注：`P0=崩溃/数据丢失` `P1=体验严重受损` `P2=体验轻微受损`
