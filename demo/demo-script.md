# ReqBot 面试演示脚本（10 分钟）

## 演示目标

向面试官展示：
1. 对 PM 工作流痛点的深刻理解
2. AI Agent 产品的系统设计能力（非 Prompt 工程）
3. Claude Code Skill 实操能力
4. 产品思维的自洽性（用 ReqBot 写 ReqBot 的 PRD）

---

## 时间线

```
0:00 ───── 2:00 ───── 5:00 ───── 8:00 ───── 10:00
  开场      实时 Demo      架构讲解      Q&A 过渡
```

---

## Part 1: 开场（0:00 - 2:00）

### 痛点共鸣（30s）

> "作为一个 PM，我每次写 PRD 都面临三个问题：
> 第一，访谈干系人时我是否问全了？有没有遗漏关键问题？
> 第二，PRD 里的需求如果后来被质疑，我能追溯到当时的原始输入吗？
> 第三，边界用例靠个人经验枚举，有没有系统性方法保证不遗漏？
>
> 所以我做了 ReqBot——一个 AI 需求分析 Agent。"

### 一句话定位（15s）

> "ReqBot 不是一个写 PRD 的聊天机器人。它是一个有 5 阶段工作流、多 Agent 协作、人在回路闸口的需求分析系统。"

### 核心差异化（45s）

展示对比表（README 中的表格，口头简述）：

> "ChatGPT 写 PRD 是单次 Prompt → 单次输出。ReqBot 做了四件不同的事：
> - 结构化访谈而不是等你追问
> - 置信度评分，模糊点不猜测而是列出解读让你选
> - 每条需求可追溯到访谈原文
> - 内置审查 Agent 自检质量"

### 过渡（15s）

> "我现在现场演示一下核心流程。"

## Part 2: 实时 Demo（2:00 - 5:00）

### 准备（提前做好）

- Claude Code 已启动，已加载 ReqBot 仓库
- 选择演示用项目：「做一个AI智能客服Agent平台」（贴合岗位、复杂度适中）

### 操作步骤

**Step 1：启动（10s）**
```
/reqbot 做一个AI智能客服Agent平台
```

**Step 2：Phase 1 — 访谈（90s）**

演示要点（不需要跑完完整 4 轮，展示 2 轮即可）：
- Agent 主动提问："这个项目的目标用户是谁？"
- 用户回答后 Agent 检测到模糊表述并追问
- 展示 Agent 的置信度追问："你说'用户体验要好'，我理解有三种解读：A)页面加载<2秒 B)核心任务3步内完成 C)遵循Material Design规范。哪个最符合你的意图？"

口头说明：
> "注意这里 Agent 不是直接猜测，而是列出 2-3 种解读让我选。这就是 Agent 的自主决策能力——它判断当前信息置信度不够，触发追问策略。"

**Step 3：跳转到产物展示（60s）**

跳过完整访谈（已提前准备好示例产物），展示：
1. `output/discover-ai-cs-agent.json` — 结构化需求提取（10 条需求 + 置信度）
2. `output/analyze-ai-cs-agent.json` — MoSCoW 分类（M7/S3/C2）+ 10 条边界用例
3. `output/prd-ai-cs-agent.md` — 完整 12 章节 PRD

> "这是实际产出的 PRD。每条功能需求都有一个 FR 编号，你可以顺着这个编号回溯到访谈时干系人的原始原话。"

**Step 4：展示追溯（20s）**

在 PRD 中随意指一条需求，展示追溯链：
> "比如 FR-003 智能转人工，它的来源标签指向 Phase 1 访谈中用户说的'AI处理80%的常见问题，剩下20%转人工。转人工触发：用户要求、连续两次回答置信度<70%、敏感词检测'——这条需求如果以后被质疑，我们知道它是怎么来的。"

## Part 3: 架构讲解（5:00 - 8:00）

### 整体架构（90s）

展示架构图（用 Mermaid 图或手绘白板）：

> "ReqBot 采用的是 Supervisor + Specialists 的多 Agent 编排模式。
>
> 编排器管理 5 阶段工作流和人在回路闸口，4 个专职 Agent 各负责一个领域。
> 这不是一个Prompt的事，而是 Agent 系统的设计——每个 Agent 有独立的输入输出契约，可以独立迭代和测试。"

### 关键设计决策（90s）

挑 3 个最有说服力的决策：

**决策 1：为什么是多 Agent？**
> "单一 Agent 带长 Prompt 的问题是：指令越多越容易冲突，出了问题难定位。多 Agent 分离关注点后，访谈质量下降我只改 Interviewer，不会影响 PRD 生成的稳定性。"

**决策 2：为什么每个阶段都有确认闸口？**
> "Agent 产品最难的不是做对的时候，而是做错的时候如何纠正。5 个阶段 4 个闸口意味着用户有 4 次机会纠正方向，而不是等到最终 PRD 出来才发现跑偏。"

**决策 3：为什么不幻觉？**
> "我的规则是：信息不足就写「待确认」，同时标注需要谁确认、确认什么。这比让 Agent 编造一个听起来合理的内容更可信，也更职业。"

## Part 4: 收尾与 Q&A 过渡（8:00 - 10:00）

### 自洽性亮点（60s）

> "还有一个我想强调的设计：ReqBot 自身的 PRD 就是用 ReqBot 的标准模板写的。你在 docs/PRD-ReqBot.md 看到的是一个自洽的证明——我用我的产品定义了我的产品。这也展示了我的产品文档标准化能力。"

### 过渡到 Q&A（60s）

> "总结一下三个关键信息：
> 1. 我理解 PM 的需求分析痛点，并且用 Agent 系统而非单次 Prompt 来解决
> 2. 我具备多 Agent 编排、人在回路、追溯系统的架构设计能力
> 3. 我能用 Claude Code 独立开发 Agent 原型
>
> 欢迎任何问题，也可以随时让我现场修改 Agent 的行为逻辑。"
